차세대 3D 공간 구축 웹 플랫폼을 위한 Zero-to-One 기술 재건축 및 실행 전략 보고서
1. 서론: 기술적 정체 구간의 진단과 전략적 피벗
현대 웹 기반 3D 애플리케이션 개발, 특히 인테리어 디자인과 같이 공간감과 심미성이 중요한 도메인에서는 기능적 구현(Functional Implementation)과 경험적 최적화(Experiential Optimization) 사이에 거대한 간극이 존재한다. 현재 'Plan2Space'로 명명된 프로젝트의 진행 상황을 담은 'working.pdf'와 관련 기술 문서들을 심층 분석한 결과, 해당 프로젝트는 전형적인 '기술적 정체 구간(Technical Plateau)'에 봉착해 있음이 드러났다. 사용자가 제기한 "작업을 수행했음에도 체감상 변화가 없고 UI가 미려하지 않다"는 문제는 코드의 단순 오류가 아니라, 렌더링 파이프라인의 부재와 결정론적이지 않은 AI 분석 방식에서 기인한 구조적 결함이다.1
기존 프로토타입은 Three.js의 기본 기능을 통해 3D 씬을 웹 브라우저에 띄우는 데는 성공했으나, 상용 서비스 수준(Commercial Grade)의 시각적 충실도(Visual Fidelity)와 사용자 경험(UX)을 제공하는 데는 실패했다. 특히 Google Gemini 모델을 활용한 도면 분석 시도가 6x4m의 더미 박스만을 생성하며 실패한 사례는, 비정형 이미지 데이터를 정밀한 3D 좌표계로 변환하는 과정에서 LLM(Large Language Model)의 확률적 특성을 통제하지 못했음을 시사한다.1
이에 본 보고서는 기존 프로젝트의 인프라(Supabase 인증, DB 스키마 등)는 유지하되, 프론트엔드의 핵심 엔진인 3D 렌더링 시스템과 AI 분석 파이프라인을 완전히 재설계하는 '하이브리드 Zero-to-One' 전략을 제안한다. 이는 Dverso Studio나 Caffe.design과 같은 하이엔드 레퍼런스의 품질을 달성하기 위해 필수적이며, 단순히 코드를 수정하는 리팩토링(Refactoring) 수준을 넘어선 아키텍처의 재구축(Reconstruction)을 의미한다.1 본 보고서는 코딩 에이전트가 수행해야 할 기술적 명세인 '마스터 개발 가이드'와 프로젝트 오너가 수행해야 할 운영적 명세인 '사용자 액션 가이드'로 나뉘어 구성된다.
2. 마스터 개발 가이드 (Master Development Guide)
수신: Autonomous Coding Agent / Lead Frontend Engineer
목표: 고품질 시각 효과와 정밀한 도면 분석을 지원하는 차세대 3D 엔진 아키텍처 구현
이 섹션은 시스템의 척추가 될 핵심 아키텍처를 정의한다. 기존의 파편화된 코드베이스를 도메인 주도 설계(DDD) 원칙에 따라 재구성하며, 최신 React 생태계의 이점을 극대화하는 방향으로 기술 스택을 확정한다.
2.1 1단계: 핵심 아키텍처 및 상태 관리 시스템 재설계
3D 웹 애플리케이션의 성능은 60fps의 고빈도 렌더링 루프(Render Loop)와 저빈도 UI 상태 업데이트를 어떻게 격리하고 동기화하느냐에 달려 있다. 기존 프로젝트의 성능 저하(30~40fps)는 이러한 상태 관리의 비효율성에서 비롯되었다.1
2.1.1 기술 스택의 확정 (The Modern Stack)
우리는 다음과 같은 기술 스택을 강제한다. 이는 선택 사항이 아니라, 목표하는 '트렌디한' 품질을 달성하기 위한 필수 조건이다.
Core Framework: Next.js 14+ (App Router)
단순한 SPA(Single Page Application)가 아닌 Next.js를 사용하는 이유는 무거운 AI 분석 로직을 서버리스 함수(Serverless Functions)로 격리하여 클라이언트의 부하를 줄이고, 초기 로딩 속도(FCP)를 최적화하기 위함이다.1
3D Engine: React Three Fiber (R3F) v8+
Three.js를 직접 명령형(Imperative)으로 다루는 방식은 복잡도가 높아질수록 유지보수가 불가능하다. R3F를 통해 3D 씬을 선언적(Declarative) 컴포넌트 단위로 관리하며, React의 생태계와 긴밀히 통합한다.2
State Management: Zustand
Redux나 Context API는 3D 렌더링 루프 내에서 불필요한 리렌더링을 유발한다. Zustand의 'Transient Updates(일시적 업데이트)' 패턴을 사용하여, 카메라 좌표나 마우스 위치와 같이 매 프레임 변하는 데이터를 React 렌더링 사이클을 우회하여 직접 3D 뷰에 주입함으로써 60fps를 보장한다.1
Physics Engine: @react-three/rapier
기존의 Cannon.js보다 성능이 월등한 WASM(WebAssembly) 기반의 Rapier를 도입한다. 이는 1인칭 시점(Walk Mode)에서 벽과의 충돌 처리, 바닥 감지 등을 끊김 없이 처리하는 데 필수적이다.3
Styling System: Tailwind CSS
Glassmorphism(유리 질감) UI를 빠르고 일관성 있게 구현하기 위해 Tailwind의 유틸리티 클래스(backdrop-blur, bg-opacity)를 적극 활용한다.4
2.1.2 도메인 주도 디렉토리 구조 (Directory Structure)
새로운 프로젝트는 기능이 아닌 도메인(영역)을 기준으로 구조화되어야 한다. 이는 코딩 에이전트가 컨텍스트를 잃지 않고 개발을 지속하는 데 중요한 가이드라인이 된다.

2.1.3 전역 상태 스토어 설계 (Global Store Architecture)
애플리케이션의 상태는 '편집 상태(Editor State)'와 '씬 데이터(Scene Data)'로 명확히 분리되어야 한다.
useEditorStore: 현재 뷰 모드(Top-view vs Walk-view), 선택된 객체 ID, UI 패널의 열림/닫힘 상태 등을 관리한다.
useSceneStore: 벽, 바닥, 가구의 좌표 및 메타데이터를 관리한다. 이 데이터는 Supabase DB와 동기화된다.
2.2 2단계: AI 분석 파이프라인의 전면 교체 (The Semantic Pipeline)
기존 Gemini 모델의 실패는 AI에게 "3D 모델을 만들어줘"라는 모호한 지시를 내렸기 때문이다. 이를 해결하기 위해 **'구조적 파싱(Semantic Parsing) 후 절차적 생성(Procedural Generation)'**이라는 2단계 접근법을 도입한다.
2.2.1 모델 변경 및 프롬프트 엔지니어링 전략
Gemini 대신 시각적 추론 능력이 뛰어난 Claude 3.5 Sonnet을 채택한다. 중요한 점은 AI가 3D 좌표를 계산하게 하는 것이 아니라, 도면의 **위상 정보(Topology)**만을 JSON으로 추출하게 하는 것이다.5
시스템 프롬프트 명세:
"당신은 건축 도면을 분석하여 CAD 데이터를 추출하는 전문가입니다. 이미지를 분석하여 다음 JSON 스키마에 맞춰 벽과 개구부(문, 창문)의 데이터를 추출하십시오.
모든 좌표는 이미지의 좌상단(0,0)을 기준으로 한 상대적 픽셀 좌표입니다.
벽(Wall)은 start와 end 점을 가진 선분으로 표현하며, 두께(thickness)를 가집니다.
개구부(Opening)는 특정 벽(wall_id)에 속하며, 벽의 시작점으로부터의 거리(offset)와 너비(width)로 정의됩니다.
가구, 텍스트, 장식 요소를 무시하고 오직 구조적 요소에만 집중하십시오."
2.2.2 Human-in-the-Loop (2D 보정 단계) 도입
AI의 인식률은 100%가 될 수 없다. 따라서 AI가 추출한 JSON 데이터를 바로 3D로 변환하는 것이 아니라, 2D 편집기를 거치도록 설계한다.
Canvas Overlay: 원본 도면 이미지 위에 AI가 추출한 벽 선(Line)과 문/창문 위치를 오버레이로 그린다.
Interaction: 사용자가 드래그하여 잘못된 벽의 위치를 수정하거나, 누락된 벽을 추가할 수 있는 인터페이스를 제공한다.
Confirmation: 사용자가 "3D 변환" 버튼을 누를 때 비로소 확정된 JSON 데이터가 3D 엔진으로 전달된다. 이 과정이 있어야만 "도면 인식이 엉망이다"라는 사용자 불만을 원천적으로 해결할 수 있다.1
2.3 3단계: 절차적 3D 생성 및 CSG 알고리즘 (The Generator)
확정된 2D JSON 데이터를 바탕으로 Three.js 상에서 3D 메쉬를 생성하는 과정은 수학적이고 절차적이어야 한다.
2.3.1 CSG(Constructive Solid Geometry)를 이용한 벽체 생성
단순히 BoxGeometry를 나열하는 방식은 창문이나 문을 뚫을 수 없어 현실감이 떨어진다. 이를 해결하기 위해 three-bvh-csg 라이브러리를 도입한다.1
알고리즘 로직:
Base Wall: 벽의 길이와 높이(2.4m)를 가진 ExtrudeGeometry를 생성한다.
Void Mesh: 문과 창문의 위치에 해당하는 크기의 '구멍 뚫기용 메쉬(Negative Mesh)'를 생성한다.
Boolean Operation: Base Wall - Void Mesh 연산을 수행하여, 물리적으로 구멍이 뚫린 벽체 메쉬를 생성한다.
이 방식은 1인칭 시점에서 창문 밖을 보거나 문을 통과하는 경험을 가능하게 하며, 빛이 개구부를 통해 들어오는 연출을 가능하게 한다.
2.3.2 바닥 및 천장 처리
Top-View 모드: 천장(Ceiling) 메쉬의 visible 속성을 false로 설정하여 내부를 볼 수 있게 한다.
Walk Mode: 천장을 렌더링하여 폐쇄된 공간감을 조성한다. 바닥(Floor)에는 타일링(Tiling) 처리가 된 PBR 텍스처를 적용하여 텍스처가 늘어지는 현상을 방지한다.
2.4 4단계: 시각적 품질의 퀀텀 점프 (The Visual Engine)
사용자가 언급한 "구리다"는 평가는 쉐이딩과 조명 처리의 미흡함에서 온다. 이를 해결하기 위해 PBR 워크플로우와 포스트 프로세싱을 전면 도입한다.
2.4.1 PBR (Physically Based Rendering) 표준화
모든 메쉬는 MeshStandardMaterial 또는 MeshPhysicalMaterial을 사용해야 한다.
HDRI (High Dynamic Range Imaging): @react-three/drei의 <Environment> 컴포넌트를 사용하여 실사 기반의 조명 정보를 씬에 적용한다. 이는 금속 재질의 가구나 유리창의 반사광을 리얼하게 표현하는 핵심 기술이다.1
Contact Shadows: 객체가 바닥에 떠 있는 느낌(Floating)을 없애기 위해 @react-three/drei의 <ContactShadows>를 사용한다. 이는 연산 비용이 높은 실시간 그림자 대신, 바닥에 닿는 부분에 부드러운 음영을 추가하여 '접지감'을 부여한다.1
2.4.2 시네마틱 포스트 프로세싱 (Post-processing)
"트렌디한" 느낌은 렌더링 후처리에서 완성된다. @react-three/postprocessing을 사용하여 다음 이펙트 체인을 구성한다.1
SSAO (Screen Space Ambient Occlusion): 벽과 벽이 만나는 모서리나 가구 틈새에 그림자를 더해 입체감을 극대화한다.
Bloom: 창문 밖의 빛이나 조명 기구에 빛 번짐 효과를 주어 몽환적이고 고급스러운 분위기를 연출한다. (임계값 조절 필수)
Noise & Vignette: 약간의 필름 그레인(Noise)과 외곽 비네팅을 추가하여, 디지털 3D 특유의 차가움을 없애고 매거진 사진과 같은 질감을 만든다.
2.5 5단계: 인터랙션 및 UX 오버레이 (The Interaction Layer)
2.5.1 이원화된 카메라 컨트롤 시스템
사용자의 요구사항인 탑뷰(배치)와 1인칭(탐험)을 완벽하게 지원하기 위해 두 가지 컨트롤러를 구현하고 상태에 따라 스위칭한다.
Edit Mode (Top-Down): OrthographicCamera와 MapControls를 사용한다. 원근감이 제거된 정투영 뷰는 가구 배치의 정확도를 높여준다.
Walk Mode (FPS): PerspectiveCamera와 PointerLockControls를 사용한다. 이때 카메라는 Rapier 물리 엔진의 RigidBody (캡슐 형태) 안에 배치되어, 사용자가 벽을 뚫고 지나가지 않도록 물리적인 충돌 처리를 수행한다.3
2.5.2 3D 팝업 및 UI 오버레이 (Spatial UI)
3D 팝업: 회원가입/로그인 창이나 가구 정보창은 단순한 HTML 모달이 아니라, 3D 공간 내에 존재하는 것처럼 보여야 한다. @react-three/drei의 <Html transform> 컴포넌트를 사용하여 UI가 3D 좌표에 고정되고 카메라 움직임에 따라 원근감이 적용되도록 구현한다.7
Glassmorphism UI: 로딩 화면과 메인 대시보드는 Tailwind CSS의 backdrop-blur-md, bg-white/10 등의 클래스를 활용하여 반투명한 유리 질감을 구현한다. 이는 뒷배경의 3D 씬이 은은하게 비치도록 하여 몰입감을 해치지 않으면서 현대적인 느낌을 준다.1
2.5.3 트렌디한 로딩 시퀀스
단순한 스피너 대신, useProgress 훅을 활용하여 로딩 진행률을 감지하고, Framer Motion을 이용해 로고가 서서히 차오르거나 3D 씬이 와이어프레임에서 솔리드 모드로 전환되는 식의 연출을 포함한다. 초기 진입 시 "Loading..." 텍스트보다는 브랜드 로고와 함께 부드러운 페이드인 효과를 적용한다.
3. 사용자 액션 가이드 (User Action Guide)
수신: Product Owner (User "숲")
목표: 개발 에이전트가 최상의 결과물을 낼 수 있도록 자산(Asset)과 데이터(Data)를 준비하고 품질을 검수하는 운영 가이드
코딩 에이전트는 코드를 짤 수 있지만, 그 안에 들어갈 '콘텐츠의 질'은 사용자가 결정해야 합니다. 다음 단계들을 직접 수행하여 에이전트에게 제공해 주십시오.
3.1 액션 1: 고품질 에셋 큐레이션 (The Asset Pipeline)
"UI가 구리다"는 느낌을 지우기 위해서는 코드보다 에셋의 퀄리티가 더 중요할 수 있습니다. 다음 소스에서 에셋을 확보하여 프로젝트의 public/ 폴더에 구조화해야 합니다.
PBR 텍스처 확보 (Poly Haven):
Poly Haven에 접속하여 'Textures' 섹션으로 이동합니다.
필수 텍스처: Wood_Floor (마루), Plaster (벽지), Concrete (모던한 느낌), Marble (주방 상판용).
다운로드 설정: 4K 해상도는 웹에서 너무 무거우므로 1K 또는 2K JPG 포맷을 다운로드하십시오. 반드시 Diffuse, Roughness, Normal, AO 맵이 모두 포함되어야 합니다.
GLB 가구 모델 확보 (Sketchfab/TurboSquid):
가구 모델은 반드시 GLB/gLTF 포맷이어야 합니다.
폴리곤 수 관리: 웹 브라우저 부하를 줄이기 위해 모델당 폴리곤 수가 10,000개 미만인 'Low Poly' 모델을 우선 선택하십시오.
다운로드: 의자, 소파, 테이블, 침대, 조명 등 카테고리별로 최소 1개씩의 고품질 샘플을 준비하십시오.1
에셋 최적화 (Draco Compression):
다운로드한 GLB 파일이 5MB를 넘어가면 로딩이 느려집니다. gltf-pipeline 도구를 사용하여 Draco 압축을 수행하거나, 온라인 압축 툴을 사용하여 용량을 1/10 수준으로 줄이십시오.
3.2 액션 2: AI 프롬프트 최적화 (The Prompt Tuning)
Claude 3.5나 GPT-4o가 도면을 정확히 인식하게 하려면, 사용자가 직접 프롬프트를 테스트하고 튜닝해야 합니다.
Claude 프롬프트 테스트: Anthropic 콘솔이나 ChatGPT에서 도면 이미지를 업로드하고 다음 프롬프트를 입력하여 결과가 JSON으로 깔끔하게 나오는지 확인하십시오.
"이 건축 도면 이미지를 분석하여 위상 정보를 추출해줘. 가구와 텍스트는 무시하고, 오직 벽(Walls)과 문/창문(Openings)의 좌표만 필요해. 좌표계는 이미지 좌상단을 (0,0)으로 하고, 출력은 오직 JSON 포맷으로만 해줘. JSON 스키마: { scale: number, walls: [{start: [x,y], end: [x,y], thickness: number}], openings: [...] }"
결과가 잘 나온다면, 이 프롬프트를 복사하여 개발 에이전트에게 "이 프롬프트를 API 로직에 사용해"라고 전달하십시오.
3.3 액션 3: 품질 검수 (QA Checklist)
개발 에이전트가 결과물을 내놓으면 다음 기준으로 냉정하게 평가하고 피드백을 주어야 합니다.
"붕 뜬 느낌" 확인: 가구가 바닥에 착 붙어 있나요, 아니면 그림자가 없어 공중에 떠 보이나요? -> 떠 보인다면 ContactShadows의 opacity와 blur 값을 높이라고 지시하십시오.
"장난감 재질" 확인: 벽이나 바닥이 너무 매끈해서 플라스틱 같나요? -> 텍스처의 Roughness 맵이 제대로 적용되었는지, 조명의 Environment 맵이 적용되었는지 확인을 요청하십시오.
"벽 뚫기" 확인: 1인칭 모드에서 벽을 뚫고 지나갈 수 있나요? -> Rapier 물리 엔진의 Collider가 벽체 생성 로직에 제대로 적용되지 않은 것입니다. 수정을 요청하십시오.
4. 결론 및 제언
현재의 프로젝트 상태는 실패가 아니라, 더 높은 단계로 도약하기 위한 과도기적 단계입니다. 본 보고서에서 제시한 '하이브리드 Zero-to-One' 전략은 기존의 백엔드 자산을 보호하면서도, 사용자가 가장 불만족스러워했던 시각적 경험과 AI의 정확도를 근본적으로 혁신할 수 있는 유일한 길입니다.
특히 CSG 기반의 절차적 벽체 생성과 PBR/포스트 프로세싱 파이프라인의 도입은 Plan2Space를 단순한 토이 프로젝트가 아닌, 상용화 가능한 수준의 공간 디자인 플랫폼으로 격상시킬 것입니다. 이제 코딩 에이전트에게 '마스터 개발 가이드'를 전달하고, 사용자는 '사용자 액션 가이드'에 따라 고품질의 재료(에셋)를 준비함으로써 프로젝트의 2막을 시작하시기 바랍니다.
